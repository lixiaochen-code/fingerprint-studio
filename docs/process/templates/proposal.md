# Proposal: <change-title>

> 模板使用说明：所有字段必填。无内容的字段填 `N/A` 并附一句话说明为什么不适用。

## 1. Intent
（为什么要做这件事，描述问题与动机）

## 2. Scope
（**做什么** + **不做什么**，分两段写）

## 3. Approach
（大方向，不写实现细节，留给 design.md）

## 4. Affected Scopes
| 范围 | 影响 |
|---|---|
| 端 | desktop / backend / shared |
| 模块 | profiles / proxies / scripts / stealth / kernel / _cross |
| 代码 | 是否动业务代码 |

## 5. Requirements
> 用 OpenSpec 风格的 Requirement + Scenario(GIVEN/WHEN/THEN) 写。至少一条 Requirement，每条至少一个 Scenario。

### Requirement: <requirement-name>
<requirement-statement>

#### Scenario: <scenario-name>
- GIVEN ...
- WHEN ...
- THEN ...

## 6. Constraints
（必须遵守的硬约束，例如不动某模块、不引入新依赖等）

## 7. Risks
| 风险 | 影响 | 缓解 |
|---|---|---|

## 8. Out of Scope
（明确**不做**的事，每项可成为后续 change）

## 9. Open Questions
> 在 proposal 阶段需要用户拍板的细节问题。回答后用户标记 approved，进入 design 环节。
> 每个问题给出 agent 的倾向 + 理由。

### Q1: <question>
- 选项 A: ...
- 选项 B: ...
- 倾向: <A/B>，理由: ...

## Conversation Log
> proposal 多轮 Q&A 的关键决策点（OpenSpec 风格的对话纪要）。

- YYYY-MM-DD | initial draft | ...
- YYYY-MM-DD | revision N | ...
